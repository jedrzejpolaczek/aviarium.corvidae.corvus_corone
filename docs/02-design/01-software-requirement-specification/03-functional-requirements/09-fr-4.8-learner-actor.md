# FR §4.8 — Learner Actor

*Enables the Learner actor to understand algorithm behaviour through visualisation, contextual explanation, Socratic guidance, genealogy, and interpretive access to completed study results.*
*Source: MANIFESTO Principles 25, 28.*
*Use Cases: UC-07, UC-08, UC-09, UC-10, UC-11.*

> **DEFERRED — Phase 4.** These requirements are defined now to close the traceability gap between
> UC-07..UC-11 and the FR layer. Implementation is planned in IMPL-044..IMPL-046 (ROADMAP Phase 4).
> No V1 code is blocked on these requirements. Each FR below is labelled `[DEFERRED]`.

---

## FR-27 `[DEFERRED]`

**The Algorithm Visualization Engine MUST produce at least one visual representation of a registered Algorithm Instance's behaviour on request — chosen from: convergence animation, parameter-sensitivity heatmap, or search-trajectory plot.**

- **Priority:** SHOULD (V1 deferred; MUST for Phase 4 release)
- **Source:** MANIFESTO Principle 25 (contextual understanding), Principle 28 (education and support)
- **Accepted when:** `visualize(algorithm_instance_id, mode="convergence")` returns a renderable artefact (image file path or in-memory figure object) without error for any registered Algorithm Instance that has at least one completed Run.
- **Connects to:** `docs/02-design/01-software-requirement-specification/02-use-cases/08-uc-07.md`; `docs/02-design/02-architecture/03-c4-leve2-containers/06-algorithm-visualization-engine.md`; IMPL-044

## FR-28 `[DEFERRED]`

**On request, the system MUST provide a contextual explanation of a specified Algorithm Instance, linking its behaviour to theoretical properties and practical examples drawn from completed benchmark results.**

- **Priority:** SHOULD (V1 deferred; MUST for Phase 4 release)
- **Source:** MANIFESTO Principle 25, Principle 28
- **Accepted when:** `explain(algorithm_instance_id, context="theoretical")` returns a non-empty structured explanation that references at least one MANIFESTO principle or published theoretical property; `explain(..., context="practical")` references at least one concrete benchmark result from a completed Experiment.
- **Connects to:** `docs/02-design/01-software-requirement-specification/02-use-cases/09-uc-08.md`; `docs/02-design/02-architecture/03-c4-leve2-containers/07-study-orchestrator.md`; IMPL-045

## FR-29 `[DEFERRED]`

**The Socratic Guide MUST respond to Learner questions about algorithm design or experimental results exclusively with bridging questions and conceptual scaffolding. It MUST NOT provide direct answers to open-ended conceptual questions.**

- **Priority:** SHOULD (V1 deferred; MUST for Phase 4 release)
- **Source:** MANIFESTO Principle 28
- **Accepted when:** A test prompt asking "What is the best algorithm for continuous optimisation?" produces a response containing at least one question directed back at the Learner and zero declarative statements asserting a single best algorithm. Verified by an LLM-as-judge that scores responses on the Socratic dimension.
- **Connects to:** `docs/02-design/01-software-requirement-specification/02-use-cases/10-uc-09.md`; `docs/02-design/02-architecture/04-c4-leve3-components/02-corvus-pilot/`; IMPL-045

## FR-30 `[DEFERRED]`

**The system MUST provide an algorithm genealogy for any registered Algorithm Instance, tracing predecessor algorithms, historical development milestones, and the design-choice rationale for each evolutionary step.**

- **Priority:** SHOULD (V1 deferred; MUST for Phase 4 release)
- **Source:** MANIFESTO Principle 25, Principle 28
- **Accepted when:** `genealogy(algorithm_instance_id)` returns a directed acyclic graph of `AlgorithmNode` objects where every node has a non-empty `rationale` field; the graph for TPE includes at least the Sequential Model-Based Optimisation lineage back to Bayesian Optimisation.
- **Connects to:** `docs/02-design/01-software-requirement-specification/02-use-cases/11-uc-10.md`; `docs/02-design/02-architecture/03-c4-leve2-containers/06-algorithm-visualization-engine.md`; IMPL-046

## FR-31 `[DEFERRED]`

**The system MUST provide Learner read-only access to completed Experiment results with interpretive support — scope statements, limitations, and open questions — and MUST NOT produce algorithm rankings or conclusions that exceed the scope of the tested Problem Instances.**

- **Priority:** SHOULD (V1 deferred; MUST for Phase 4 release)
- **Source:** MANIFESTO Principle 3 (scope of claims), Principles 23–25 (reporting), Principle 28
- **Accepted when:** (1) A Learner-mode `explore(experiment_id)` call raises `PermissionError` on any write or delete operation against the Results Store. (2) The interpretive summary returned always includes a non-null `scope_statement` and `open_questions` field. (3) No ranking of algorithms appears in the Learner-mode output — verified by the same constraint enforcement test as FR-21.
- **Connects to:** `docs/02-design/01-software-requirement-specification/02-use-cases/12-uc-11.md`; `docs/02-design/02-architecture/03-c4-leve2-containers/05-reporting-engine.md`; `docs/02-design/02-architecture/03-c4-leve2-containers/12-results-store.md`; IMPL-044..046
