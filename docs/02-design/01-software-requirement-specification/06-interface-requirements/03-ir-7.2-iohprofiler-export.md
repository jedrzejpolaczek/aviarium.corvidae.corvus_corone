# §7.2 — IOHprofiler Export Interface

**Direction:** Export (Corvus Corone → IOHprofiler)
**Format:** IOHprofiler v0.3.3+ — `.dat` performance log + `.json` sidecar per (algorithm, function) pair
**Protocol:** Local filesystem write; files are consumed by IOHanalyzer offline
**Requirement:** FR-24
**Data mapping:** [docs/03-technical-contracts/01-data-format/11-interoperability-mappings.md §4.2](../../../03-technical-contracts/01-data-format/11-interoperability-mappings.md)
**Implementation:** `corvus_corone.export.ioh_exporter.IOHExporter`
**Status:** ✅ Mapping complete (REF-TASK-0007)

---

## Output Layout

```
<output_dir>/
└── <algorithm_id>/
    ├── IOHprofiler_f<func_id>_<func_name>.json   ← JSON sidecar
    └── data_f<func_id>_<func_name>/
        └── IOHprofiler_f<func_id>_DIM<dim>.dat   ← performance log
```

## Field Mapping Summary

**`.dat` file** (two-column, unquoted header repeated at each run boundary):

| Column | Source field | Notes |
|---|---|---|
| `evaluations` | `PerformanceRecord.evaluation_number` | 1-based integer |
| `raw_y` | `PerformanceRecord.best_so_far` (or `objective_value`) | Best-so-far value at this evaluation |

**JSON sidecar** stores fields with no `.dat` equivalent:

| Sidecar field | Source | Notes |
|---|---|---|
| `function_id` | `ProblemMeta.func_id` | Caller-supplied; auto-assigned if absent (LOSS-IOH-06) |
| `function_name` | `ProblemMeta.func_name` | |
| `maximization` | `ProblemMeta.maximization` | Tells IOHanalyzer ECDF orientation |
| `algorithm.name` | `AlgorithmMeta.name` | |
| `corvus_seed` | `RunRecord.seed` | Non-standard extension; ignored by IOHanalyzer |
| `corvus_run_id` | `RunRecord.run_id` | Non-standard extension; ignored by IOHanalyzer |

Full field-level mapping and all manifest keys (LOSS-IOH-01 through LOSS-IOH-09) are in
[§4.2 of the interoperability mappings](../../../03-technical-contracts/01-data-format/11-interoperability-mappings.md).
File format details are in [§3.1 of the file formats spec](../../../03-technical-contracts/01-data-format/10-file-formats.md).

## Behavior

- One `.dat` file is produced per (algorithm, problem, dimension) combination; all runs are concatenated, separated by a repeated header line (run boundary marker)
- IOHanalyzer auto-detects the v0.3.3+ format when the `.json` sidecar is present
- Every export call returns an information-loss manifest; the manifest is never empty (NFR-INTEROP-01)
- Fields without an IOHprofiler equivalent (`elapsed_time`, `trigger_reason`, `seed`, `run_id`) are listed in the manifest
- Runs with `status = "failed"` are excluded from the `.dat` file and flagged in the manifest (LOSS-IOH-05)

## Failure Handling

- Export is rejected if `PerformanceRecord.evaluation_number` is absent (mandatory `.dat` column has no source)
- `ProblemMeta` and `AlgorithmMeta` are optional; missing entries receive auto-generated defaults with manifest warnings (LOSS-IOH-06, LOSS-IOH-07)
- Partially written files are left on disk on unexpected failure; the caller is responsible for cleanup (unlike COCO, no archive format is involved)
