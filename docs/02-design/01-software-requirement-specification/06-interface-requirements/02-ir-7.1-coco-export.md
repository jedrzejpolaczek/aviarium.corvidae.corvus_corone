# §7.1 — COCO Export Interface

**Direction:** Export (Corvus Corone → COCO)
**Format:** BBOBNewDataFormat — `.dat` performance log + `.info` index file per (algorithm, function) pair
**Protocol:** Local filesystem write; files are consumed by `cocopp` post-processing tools offline
**Requirement:** FR-23
**Data mapping:** [docs/03-technical-contracts/01-data-format/11-interoperability-mappings.md §4.1](../../../03-technical-contracts/01-data-format/11-interoperability-mappings.md)
**Status:** ✅ Mapping complete (REF-TASK-0005)

---

## Field Mapping Summary

| `.dat` column | Source field | Notes |
|---|---|---|
| col 0 | `PerformanceRecord.evaluation_number` | Function evaluations, 1-based integer |
| col 1 | `0` (hard-coded) | Constraint evaluations; always 0 (Corvus has no constraint support in V1) |
| col 2 | `best f(x) − f_opt` so far | Computed from `known_optimum` when available; `objective_value` used as proxy otherwise (LOSS-COCO-01 **critical**) |
| col 3 | `PerformanceRecord.objective_value` | Current f(x) |

`.info` index fields: `funcId`, `DIM`, `Precision`, `algId` (= `algorithm_id`), `comment` (= algorithm name).

Full field-level mapping and all manifest keys (LOSS-COCO-01 through LOSS-COCO-10) are in
[§4.1 of the interoperability mappings](../../../03-technical-contracts/01-data-format/11-interoperability-mappings.md).

## Behavior

- The system converts a completed Experiment's Performance Records to BBOBNewDataFormat
- `f(x) − f_opt` gap is computed exactly when `ProblemInstance.objective.known_optimum` is non-null
- When `known_optimum` is null, `objective_value` is used as a proxy and LOSS-COCO-01 (severity: **critical**) is appended to the information-loss manifest
- Every export call returns an information-loss manifest; the manifest is never empty (NFR-INTEROP-01)
- Fields without a COCO equivalent (e.g., `seed`, `run_id`, `elapsed_time`) are listed in the manifest with severity `informational`
- The produced files MUST be loadable by `cocopp` post-processing tools without modification

## Failure Handling

- If a mandatory COCO field has no source in the Corvus Corone schema (e.g., missing `evaluation_number`), the export is rejected with an explicit error identifying the missing field
- Partially written archives MUST be cleaned up on failure; no corrupt partial exports are left on disk
- Information-loss items with severity `critical` are surfaced prominently in the manifest but do **not** cause the export to fail — the caller decides whether to proceed
